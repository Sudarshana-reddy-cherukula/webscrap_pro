### Technical Debt

1. **`backend/.env` committed to git** — real credentials exposed. Even though `.gitignore` lists `.env`, the file is tracked.
2. **Python services have empty directories** (`routes/`, `scrapers/`, `utils/`, `services/`) — all logic crammed into `app.py`.
3. **Docker Compose references `pdf-service/` and `scraping-service/` at root** but actual code is in `backend/python-services/`.
4. **Both `pdfService` and `pdfApi` existed** with different method sets — now consolidated.
5. **Backend `package.json` has `"lint": "eslint ."`** but no `.eslintrc` config for the backend.

### Scalability Problems

1. **Synchronous processing** — scraping and PDF operations block the event loop. A single large scrape can starve all other requests.
2. **No connection pooling config** — MongoDB Atlas connection has no pool size limits.
3. **No caching layer** — every dashboard load, analytics query, and job list hits MongoDB directly.
4. **In-memory Socket.io** — won't work across multiple server instances. Needs Redis adapter.
5. **No horizontal scaling story** — sessions are in-memory via JWT (good), but file uploads are local disk.

### Security Issues

1. **Real `.env` in git history** — even if removed now, credentials are in git history.
2. **No CSRF protection** — relying solely on JWT in headers (acceptable for API-only, but risky if cookies are ever used).
3. **Rate limiting is global** — no per-user or per-endpoint rate limits.
4. **No input sanitization** on scraping URLs — potential SSRF.
5. **JWT secret is a simple string** — should be a 256-bit random value.
6. **No `httpOnly` cookie option** — token stored in `localStorage` (XSS vulnerable).

### Performance Issues

1. **No response caching** — repeated identical queries hit the database every time.
2. **No compression** — `compression` middleware exists but effectiveness unknown without proper headers.
3. **Large bundle risk** — Puppeteer + Playwright + Cheerio all loaded in the same backend process.
4. **No CDN** — static assets served directly from Vercel (acceptable) but API has no edge caching.
5. **PDF processing is CPU-intensive** — needs worker threads or separate service.

### Deployment Issues

1. **Render free tier** — cold starts of 30-60s. Backend sleeps after inactivity.
2. **Vercel frontend + Render backend** — CORS complexity, different regions possible.
3. **No staging environment** — every deploy goes to production.
4. **No database migrations** — schema changes require manual coordination.
5. **No backup strategy** — MongoDB Atlas backups depend entirely on Atlas tier.

### API Design (Current)

- All routes under `/api/` with no versioning
- Inconsistent response formats: some return `{ data: { ... } }`, others return data directly
- No pagination cursors — using offset-based pagination
- No rate limiting per endpoint
- No request ID tracking

### Database Design (Current)

- 6 models: User, ScrapeJob, PDFJob, Export, ApiKey, Session
- No indexes beyond Mongoose defaults
- No TTL on sessions
- No soft deletes
- No audit trail
- No composite indexes for common queries

### Frontend Architecture (Current)

- React 19 with lazy loading (good)
- No global state management
- Context-based auth (acceptable for this scale)
- Theme system exists but is broken/no-op
- No React Query/SWR for server state
- No optimistic updates
- No offline support
- No PWA manifest

### Backend Architecture (Current)

- Express MVC pattern
- No service layer separation (controllers call services directly, which is OK)
- No repository pattern
- No dependency injection
- No graceful shutdown handling
- No request context (correlation IDs)
- Socket.io for real-time (good but not scaled)

### Controllers

```javascript
// Pattern: Controller → Service → Repository → Model
// Controllers handle: HTTP parsing, validation, response formatting
// Controllers do NOT: business logic, DB queries

// Example: scrapeController.js
class ScrapeController {
  constructor(scrapeService, cacheService) {
    this.scrapeService = scrapeService
    this.cacheService = cacheService
  }

  async createJob(req, res, next) {
    try {
      const { urls, options, scrapeType } = req.validatedBody  // Pre-validated by Zod middleware
      const job = await this.scrapeService.createJob(req.user.id, { urls, options, scrapeType })
      
      // Emit real-time update
      req.io.to(`user:${req.user.id}`).emit('job:created', job)
      
      res.status(201).json({
        success: true,
        data: { job },
        message: 'Scrape job created'
      })
    } catch (error) {
      next(error)  // Pass to global error handler
    }
  }

  async getJobStatus(req, res, next) {
    try {
      const { jobId } = req.params
      const cacheKey = `job:${jobId}:status`
      
      let status = await this.cacheService.get(cacheKey)
      if (!status) {
        status = await this.scrapeService.getJobStatus(jobId, req.user.id)
        await this.cacheService.set(cacheKey, status, 30)  // 30s cache
      }
      
      res.json({ success: true, data: status })
    } catch (error) {
      next(error)
    }
  }
}
```

```javascript
    this.scrapeQueue = scrapeQueue
    this.cacheService = cacheService
    this.analyticsService = analyticsService
  }

  async createJob(userId, { urls, options, scrapeType }) {
    // Check rate limits
    await this.checkJobLimits(userId)
    
    // Create DB record
    const job = await this.scrapeJobRepo.create({
      userId,
      targetUrls: urls,
      scrapeType,
      options,
      status: 'queued'
    })
    
    // Enqueue for processing
    const queueJob = await this.scrapeQueue.add('process-scrape', {
      jobId: job._id,
      userId,
      urls,
      options,
      scrapeType
    }, {
      priority: options.priority || 0,
      attempts: 3,
      backoff: { type: 'exponential', delay: 5000 }
    })
    
    // Update with queue reference
    await this.scrapeJobRepo.update(job._id, { queueId: queueJob.id })
    
    // Track analytics
    await this.analyticsService.track(userId, 'scrape_created', { scrapeType, urlCount: urls.length })
    
    return job
  }

  async checkJobLimits(userId) {
    const user = await this.userRepo.findById(userId)
    const activeJobs = await this.scrapeJobRepo.countActiveByUser(userId)
    
    if (activeJobs >= user.limits.maxConcurrentJobs) {
      throw new RateLimitError('Maximum concurrent jobs reached')
    }
    
    if (user.usage.scrapesThisMonth >= user.limits.maxScrapesPerMonth) {
      throw new UsageLimitError('Monthly scrape limit reached')
    }
  }
}
```

```javascript
function errorHandler(err, req, res, _next) {
  const correlationId = req.correlationId
  
  if (err.isOperational) {
    logger.warn({ err, correlationId }, err.message)
    return res.status(err.statusCode).json({
      success: false,
      error: { code: err.code, message: err.message, details: err.details }
    })
  }
  
  // Unexpected error
  logger.error({ err, correlationId }, 'Unexpected error')
  Sentry.captureException(err)

    success: false,
    error: { code: 'INTERNAL_ERROR', message: 'Something went wrong' }
```

```javascript
const scrapeWorker = new Worker('scrape-queue', async (job) => {

      completedAt: new Date(),
      actualDuration: Date.now() - job.timestamp
    })

    return results

      error: { message: error.message, code: error.code }
```

```javascript
import { useQuery, useMutation, useQueryClient } from '@tanstack/react-query'
import { scraperService } from '@/services/scraperService'

export function useJobs(filters) {
  return useQuery({
    queryKey: ['jobs', filters],
    queryFn: () => scraperService.getJobs(filters.limit, filters.offset, filters.status),
    staleTime: 30000,  // 30s
    refetchInterval: (query) => {
      const jobs = query.state.data?.data?.jobs || []
      const hasActive = jobs.some(j => ['pending', 'queued', 'running'].includes(j.status))
      return hasActive ? 5000 : false  // Poll every 5s for active jobs
    }
  })
}

      queryClient.invalidateQueries({ queryKey: ['dashboard'] })
    }
  })
```

### Browser Pool Management

```javascript
// playwright pool - reuse browser instances
class BrowserPool {
  constructor(maxSize = 3) {
    this.pool = []
    this.maxSize = maxSize
  }
  
  async acquire() {
    if (this.pool.length > 0) return this.pool.pop()
    if (this.pool.length < this.maxSize) {
      return await chromium.launch({ headless: true })
    }
    // Wait for an instance to become available
    return new Promise(resolve => this.waitQueue.push(resolve))
  }
  
  release(browser) {
    if (this.waitQueue.length > 0) {
      this.waitQueue.shift()(browser)
    } else {
      this.pool.push(browser)
    }
  }
}
```

```javascript
async function detectDuplicates(content) {

  const candidates = await findSimilarFingerprints(fingerprint, threshold = 0.9)
```

```javascript
          message: i.message
        }))
      }
    })
  }
  
  req.validatedBody = result.data.body
  req.validatedQuery = result.data.query
  next()
}
```

## 18. Interview Questions

### Technical Questions

**Q: How would you handle scraping a website that blocks automated requests?**

> A: Multi-layered approach: (1) Rotate residential proxies to avoid IP blocking, (2) Randomize browser fingerprints (user agent, viewport, plugins), (3) Implement human-like delays between requests (randomized 2-5s), (4) Use Playwright over Puppeteer for better anti-detection, (5) Maintain session cookies, (6) Detect CAPTCHA and switch strategies, (7) Respect robots.txt to avoid aggressive crawling, (8) Use headful mode with virtual display (Xvfb) if needed.

**Q: How do you ensure data consistency in a distributed queue system?**

> A: (1) BullMQ uses Redis atomic operations for job state transitions, (2) MongoDB transactions for multi-document operations, (3) Idempotency keys on job creation to prevent duplicates, (4) Distributed locks via Redis for concurrent access to same resources, (5) Dead letter queues for failed jobs, (6) Job deduplication based on content hash.

**Q: Explain the difference between horizontal and vertical scaling in your architecture.**

> A: Vertical (scale up): Upgrade Render instance to more CPU/RAM — simple but has limits. Horizontal (scale out): Add more backend/worker instances behind load balancer — requires stateless design, Redis for shared state, MongoDB replica sets. Our architecture uses Redis for session/cache/queues (shared state), JWT for auth (stateless), and BullMQ for distributed job processing — making horizontal scaling straightforward.

### System Design Questions

**Q: Design the real-time job progress system.**

> A: (1) Client connects via Socket.io, joins a room based on userId, (2) Worker emits progress events during job processing, (3) Backend broadcasts to user's room, (4) Client updates UI in real-time, (5) Fallback: SSE (Server-Sent Events) for clients that can't use WebSocket, (6) Redis adapter for Socket.io to scale across multiple server instances, (7) Message queue (BullMQ events) as backup for missed WebSocket messages.

**Q: How would you design the caching strategy?**

> A: Three-tier caching: (1) Client-side: TanStack Query with staleTime + refetchInterval for active jobs, (2) API-level: Redis with TTL per endpoint type — dashboard stats (5min), analytics (1min), user profile (10min), job lists (30s), (3) Database: MongoDB Atlas built-in query cache + index optimization. Cache invalidation: event-driven (job complete → invalidate dashboard), pattern-based (user:* on profile change), TTL-based (automatic expiry).

### Backend Questions

**Q: Why did you choose BullMQ over alternatives like RabbitMQ?**

> A: (1) BullMQ is Redis-native — we already use Redis for caching, no additional infrastructure, (2) Better Node.js integration (same language), (3) Built-in rate limiting, retries, priorities, delayed jobs, (4) Better monitoring UI (Bull Board), (5) Simpler deployment than RabbitMQ, (6) Good enough for our throughput needs (< 1000 jobs/minute). Trade-off: Redis is single-threaded for queue operations, but at our scale this is fine.

**Q: How do you handle file uploads securely?**

> A: (1) Multer with size limits (10MB), (2) MIME type validation (only PDF allowed for PDF endpoints), (3) Rename files to prevent path traversal, (4) Store in S3/R2 not local disk (production), (5) Generate signed URLs for downloads (time-limited), (6) Virus scanning on upload (future), (7) Cleanup old files via cron job, (8) User-level storage quotas.

### React Questions

**Q: Why TanStack Query over just using useEffect + useState for data fetching?**

> A: (1) Automatic caching — same query across components shares cache, (2) Background refetching — stale data updated automatically, (3) Optimistic updates — UI updates immediately, rolls back on error, (4) Deduplication — multiple components requesting same data make one network call, (5) Pagination/infinite scroll built-in, (6) Request cancellation on unmount, (7) Retry logic with exponential backoff, (8) DevTools for debugging cache state.

**Q: How do you handle authentication state across the app?**

> A: React Context for auth state (AuthProvider wraps app), TanStack Query for server-side auth data (profile, settings). Token in memory (not localStorage for XSS safety), refresh token in httpOnly cookie. AuthContext provides: user, token, login, logout, setUser. On mount: check if token exists → fetch profile → update context. On 401: auto-refresh token → retry request → if refresh fails, logout.

### MongoDB Questions

**Q: How do you optimize MongoDB queries for large collections?**

> A: (1) Compound indexes for common query patterns: `{ userId: 1, createdAt: -1 }` for user's jobs, (2) Partial indexes for filtered queries: `{ status: 1 }` where `status != 'completed'`, (3) TTL indexes for auto-cleanup: sessions, temp data, (4) Use `.lean()` for read-only queries (no Mongoose overhead), (5) Cursor-based pagination instead of skip/limit for large offsets, (6) Aggregation pipelines for analytics instead of fetching all documents, (7) Atlas Search for full-text search instead of regex queries.

### Authentication Questions

**Q: Why use both access tokens and refresh tokens?**

> A: Access tokens (15min) are short-lived — if compromised, damage is limited. Refresh tokens (7d, httpOnly cookie) are used to get new access tokens without re-authentication. The refresh token is only sent over HTTPS in httpOnly cookies (not accessible to JavaScript), making XSS attacks ineffective. Token rotation on refresh means each refresh token is single-use — if a stolen refresh token is used, the legitimate user's token is invalidated, alerting them to the breach.

### DevOps Questions

**Q: How do you handle zero-downtime deployments?**

> A: (1) Blue-green deployment: new version starts alongside old, health check passes, traffic switches, (2) Render supports this natively with "rolling" deploys, (3) Database migrations are backward-compatible (additive only), (4) Feature flags for gradual rollout, (5) Health check endpoint confirms readiness before receiving traffic, (6) Graceful shutdown: finish in-flight requests before terminating old instance.

**Q: Describe your monitoring and alerting strategy.**

> A: Three pillars: (1) Metrics (Prometheus/Grafana) — request rate, error rate, latency, queue depth, CPU/memory, (2) Logs (Pino → structured JSON → centralized) — correlation IDs for request tracing, (3) Errors (Sentry) — real-time error tracking with stack traces. Alerts: High error rate (>5% for 5min), high latency (p95 > 2s), queue backlog (>100 jobs), memory pressure (>80%), disk usage (>85%). Delivery: Slack webhook for warnings, PagerDuty for critical.

### AI Questions

**Q: How would you implement semantic search on scraped content?**

> A: (1) On scrape completion, chunk content into paragraphs, (2) Generate embeddings using OpenAI text-embedding-3-small, (3) Store embeddings in MongoDB Atlas with vector index, (4) On search: generate query embedding, run $vectorSearch aggregation with cosine similarity, (5) Return ranked results with similarity scores, (6) Hybrid search: combine vector search with Atlas full-text search for best results, (7) Cache frequent search results in Redis.
